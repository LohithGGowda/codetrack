Concepts You’ve Learned (Git & GitHub)

Week 3 Learning Summary (Git & GitHub) – in simple words:

  • What Git is (analogy): Git is like a video game's save system. Every time I commit, it’s like creating a new “save point” that I can go back to later.
  • Repository: A repository (repo) is the project folder where Git tracks files and their history.
  • git init: Starts Git tracking in an existing folder by creating a hidden .git database.
  • git clone: Downloads a copy of an existing repo (for example, from GitHub) to my system so I can work on it.
  • Working directory: This is where I edit files normally (like typing in HTML/CSS).
  • git status: Shows what I changed and what Git is currently tracking. It’s like a checklist of “what is modified / what is ready”.
  • Staging area (analogy): The staging area is like a shopping cart. I choose which changes I want to include using git add. I don’t have to buy everything I touched.
  • git add: Adds selected changes to staging so they will be included in the next commit.
  • Good commit messages: I should write short, clear messages like “Add contact page link” so future debugging is easier.
  • Commit (analogy): A commit is like a photograph/snapshot of the project at a point in time with a message describing what I did.
  • git commit -m: Saves the staged snapshot into Git history with a short message (example: “Fix footer link”).
  • git log: Shows the timeline of commits (who changed what and when). This helps when something breaks.
  • git diff: Shows the exact line-by-line changes between versions (useful for finding mistakes quickly).
  • .gitignore: A file that tells Git which files/folders NOT to track (like node_modules or secret config files).
  • Branching (analogy): A branch is like making a photocopy of your notebook to try a new solution. The original stays safe. I can create a branch, work there, and merge later.
  • git branch / git switch: Lets me create and move between branches (so my experiments don’t disturb the main work).
  • Merge conflict (analogy): A conflict happens when Git cannot decide between two edits (like two people editing the same sentence differently). I must manually choose the correct final version.
  • Merging: Merge combines my branch changes back into the main branch after I confirm everything is correct.
  • GitHub: GitHub stores my repo online, so I can back up code, share it, and collaborate.
  • Remote (analogy): A remote is like a cloud copy of my project. My laptop has local commits; GitHub has the remote commits.
  • git remote -v: Shows which GitHub URL my project is connected to.
  • Push/Pull: Push uploads my commits to GitHub; pull brings new changes from GitHub to my computer.
  • git fetch vs git pull: fetch only downloads updates; pull downloads and merges them into my current branch.
  • Pull Request (PR): A PR is a request to merge changes. It allows review and discussion before merging (good for team work and avoiding bugs).
  • git stash: Temporarily saves my unfinished work without committing, so I can switch branches safely and come back later.
  • git reset vs git revert: reset moves my branch pointer back (can rewrite history); revert creates a new commit that “undoes” changes (safer on shared repos).
  • Tags/Releases: A tag is like a label on a specific commit (example: v1.0) so I can easily find important versions.
  • Debugging scenario (real world): If the website suddenly stops working after changes, I can run git log to find the last commit, use git diff to see what exactly changed, and then fix it or revert the bad commit.
